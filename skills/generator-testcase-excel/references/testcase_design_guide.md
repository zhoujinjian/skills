# 测试用例设计方法论参考

本文档为测试用例生成技能提供方法论参考，重点阐述从测试点到可执行测试用例的转化规则。

---

## 一、等价类划分 → 测试用例转化

### 转化规则
1. 每个有效等价类 → 至少生成1条正向测试用例
2. 每个无效等价类 → 单独生成1条异常测试用例（不可合并无效等价类）
3. 多个输入字段的等价类组合 → 使用正交法或判定表精简组合

### 具体示例

**输入项：密码（8-20位，含字母和数字）**

| 等价类 | 类型 | 示例值 | 对应用例 |
|--------|------|--------|---------|
| 8-20位字母+数字 | 有效 | `Abc12345` | 正向用例 |
| 纯数字8位 | 无效 | `12345678` | 异常用例-缺少字母 |
| 纯字母8位 | 无效 | `abcdefgh` | 异常用例-缺少数字 |
| 7位字母+数字 | 无效 | `Abc1234` | 异常用例-长度不足 |
| 含空格 | 无效 | `Abc 1234` | 异常用例-非法字符 |
| 含中文 | 无效 | `Abc密码12` | 异常用例-非法字符 |

## 二、边界值分析 → 测试用例转化

### 转化规则
对每个输入字段的边界，必须生成以下5类测试用例：

1. **最小值** - 刚好满足下限的值
2. **略高于最小值** - 最小值+1（或+1个步长）
3. **正常值** - 范围中间的典型值
4. **略低于最大值** - 最大值-1（或-1个步长）
5. **最大值** - 刚好满足上限的值

另外，补充2条异常用例：
6. **低于最小值** - 最小值-1
7. **高于最大值** - 最大值+1

### 具体示例

**输入项：手机号（11位数字）**

| 边界位置 | 输入值 | 预期结果 | 用例类型 |
|---------|--------|---------|---------|
| 低于最小值 | 10位：1381234567 | 提示格式错误 | 异常 |
| 最小值 | 11位：13812345678 | 接受并正常处理 | 正向 |
| 略高于最小值 | 仍为11位（离散值） | 接受 | 正向 |
| 正常值 | 11位：13912345678 | 接受并正常处理 | 正向 |
| 略低于最大值 | 仍为11位（离散值） | 接受 | 正向 |
| 最大值 | 11位：19999999999 | 接受并正常处理 | 正向 |
| 高于最大值 | 12位：138123456789 | 提示格式错误 | 异常 |

**输入项：密码长度（8-20位）**

| 边界位置 | 输入值 | 预期结果 | 用例类型 |
|---------|--------|---------|---------|
| 低于最小值 | 7位：`Abc1234` | 提示密码不符合要求 | 异常 |
| 最小值 | 8位：`Abc12345` | 接受 | 正向 |
| 略高于最小值 | 9位：`Abc123456` | 接受 | 正向 |
| 正常值 | 12位：`Abc123456789` | 接受 | 正向 |
| 略低于最大值 | 19位：`Abc123456789012345` | 接受 | 正向 |
| 最大值 | 20位：`Abc1234567890123456` | 接受 | 正向 |
| 高于最大值 | 21位：`Abc12345678901234567` | 提示密码不符合要求 | 异常 |

## 三、测试步骤编写规范

### 格式要求
- 按操作顺序编号：1. 2. 3. ...
- 每步描述一个明确的操作动作
- 包含操作对象、操作方式、输入内容
- 避免模糊描述（如"输入有效数据"、"进行正常操作"）

### 正确示例
```
1. 打开注册页面
2. 在手机号输入框输入 13812345678
3. 点击"获取验证码"按钮
4. 在验证码输入框输入 123456
5. 在密码输入框输入 Abc12345
6. 点击"提交注册"按钮
```

### 错误示例
```
1. 进入注册页面
2. 输入有效的手机号
3. 获取验证码
4. 输入验证码
5. 输入密码
6. 点击注册
```

## 四、测试数据编写规范

### 格式要求
- 提供具体的测试数据值，不用"有效数据""合法值"等模糊描述
- 数据与步骤中的操作一一对应
- 对于下拉选择、枚举类型，明确具体选项值

### 正确示例
| 输入项 | 测试数据 |
|-------|---------|
| 手机号 | 13812345678 |
| 验证码 | 123456 |
| 密码 | Abc12345 |

### 错误示例
| 输入项 | 测试数据 |
|-------|---------|
| 手机号 | 任意11位手机号 |
| 验证码 | 正确的验证码 |
| 密码 | 符合要求的密码 |

## 五、预期结果编写规范

### 格式要求
- 包含页面行为验证（跳转、提示信息、按钮状态）
- 包含数据状态验证（数据库、接口返回）
- 包含UI状态验证（元素显示/隐藏、高亮、禁用）
- 异常用例必须验证提示信息的具体内容

### 正确示例
```
1. 页面跳转到首页
2. 顶部导航栏显示用户昵称
3. 提示"注册成功"
4. 用户状态为已登录
5. 数据库新增一条用户记录，手机号为13812345678
```

### 错误示例
```
注册成功
```

## 六、优先级判定标准

| 优先级 | 判定标准 | 适用场景 |
|-------|---------|---------|
| P0 | 阻塞主流程、影响核心功能、无替代方案 | 核心功能正向流程、安全漏洞、数据丢失风险 |
| P1 | 重要功能异常、有替代方案、影响用户体验 | 分支流程、边界值、常见异常场景 |
| P2 | 边缘场景、UI细节、低概率异常 | 极端边界值、罕见异常、兼容性细节 |

## 七、XMind 测试点文件解析规则

当用户上传 XMind 文件作为输入时，按以下规则解析测试点：

### XMind 文件结构
XMind 文件本质是 ZIP 压缩包，包含 `content.json` 文件。

### 解析流程
1. 解压 ZIP 文件，读取 `content.json`
2. 从 `rootTopic` 开始遍历思维导图结构
3. 识别一级节点中的分类名称（功能测试/边界值测试/异常测试/业务规则测试/非功能性测试）
4. 分类下的子节点为测试点，提取：
   - `title` → 测试点描述（可能含编号前缀如 `TC-001 | 描述`）
   - `markers` → 优先级（priority-1=P0, priority-2=P1, priority-3=P2）
   - `labels` → 关联需求编号
   - `notes` → 详细说明
   - 子节点 `children` → 子测试点
5. 将解析后的测试点转化为完整测试用例

### 从测试点到测试用例的转化
每个测试点需展开为1条或多条测试用例：
- 功能类测试点 → 通常1条正向用例
- 边界值类测试点 → 按边界值分析规则拆分为多条用例（每条用例对应一个具体边界值）
- 异常类测试点 → 通常1条异常用例（包含具体错误输入和提示信息验证）
- 业务规则类测试点 → 根据规则复杂度拆分
- 非功能性类测试点 → 按安全/性能/兼容性拆分
