# 白名单规则 — 数据清理保护清单

## 永久保护项（不可清理）

### 1. 管理员账号
- 表: `user`
- 条件: `username = 'admin'` 或 `role = 1` 的系统管理员账号
- 说明: 系统内置 admin 管理员属于常驻账号，任何情况下不做删除

### 2. 人工预录的正式数据
- 判定依据:
  - 数据内容具备明确业务含义（如真实商品、正式分类）
  - 字段中不包含 test/测试/temp/logout/pwdtest/_init 等测试标识
  - 由人工预先录入的数据（创建时间早于自动化测试开始时间）
- 保护方式: 跳过，不做清理

### 3. 系统配置数据
- 表: `captcha_config`
- 说明: 验证码配置等系统级参数永久保留

### 4. 系统缓存
- Redis key: 不清理 `captcha_config` 等系统配置相关缓存

## 可清理项（测试数据特征）

### 用户数据
- `username` 包含: `test`, `测试`, `logout_`, `pwdtest_`, `_init_`, `temp_`, `newuser_`, `bypass_`, `nonexistent`
- 注册时间在测试执行期间

### 商品数据
- `name` 包含: `测试商品`, `test_`
- 非人工预录的测试创建商品

### 分类数据
- `name` 包含: `测试分类`
- 测试执行期间创建的分类

### 订单数据
- 关联用户为测试用户的订单
- 测试执行期间创建的订单

### 地址数据
- `name` 包含: `测试收货人`, `张三` (测试模板地址)
- 关联用户为测试用户的地址

### 购物车数据
- 关联用户为测试用户的购物车记录

### 验证码记录
- 表: `captcha_record`
- 已过期或已使用的验证码记录

### Redis 缓存
- `captcha:*` 前缀的验证码缓存
- 测试用户的登录 Token 缓存

### 本地文件
- `logs/` 目录下超过 1 天的日志文件
- `allure-results/` 目录下的历史报告
- `.pytest_cache/` 缓存目录

## 存疑处理规则

当数据无法明确判定来源时：
1. 标记为「待确认」，不直接删除
2. 在清理报告中列出待确认数据的关键信息（ID、名称、创建时间）
3. 等待人工确认后再处理
