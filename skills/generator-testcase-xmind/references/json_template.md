# JSON 数据模板参考

本文档定义了测试点 JSON 数据的结构规范，用于 XMind 生成脚本的输入。

---

## 完整结构定义

```json
{
    "title": "项目名称 - 测试点分析",
    "user_stories": [
        {
            "id": "US-001",
            "title": "用户故事标题",
            "description": "作为...我希望...以便..."
        }
    ],
    "categories": [
        {
            "name": "分类名称",
            "test_points": [
                {
                    "id": "TC-001",
                    "description": "测试点描述",
                    "user_story_id": "US-001",
                    "priority": "P0",
                    "sub_points": [
                        "子测试点1",
                        "子测试点2",
                        {
                            "description": "带详情的子测试点",
                            "priority": "P1",
                            "detail": "详细说明",
                            "sub_points": ["更深层的子点"]
                        }
                    ]
                }
            ]
        }
    ]
}
```

## 五大分类标准名称

| 编号 | 分类名称 | 说明 |
|-----|---------|------|
| 1 | 功能测试 | 正常流程、分支流程测试点 |
| 2 | 边界值测试 | 输入长度、数值范围、字符类型等边界测试点 |
| 3 | 异常测试 | 错误输入、系统异常、网络异常等测试点 |
| 4 | 业务规则测试 | 业务逻辑约束、状态转换规则测试点 |
| 5 | 非功能性测试 | 性能、安全性、兼容性测试点 |

## 优先级标注规范

| 优先级 | 标签 | 图标标记 |
|-------|------|---------|
| P0 | 阻塞主流程 | 红色旗帜 |
| P1 | 重要功能异常 | 橙色旗帜 |
| P2 | 边缘场景 | 蓝色旗帜 |

## 编号规则

- 用户故事：`US-001`, `US-002`, ...
- 测试点：`TC-001`, `TC-002`, ...
- 编号全局连续，不按分类重新编号
