# 黑盒测试方法论参考

本文档为测试用例生成技能提供方法论支撑，涵盖核心测试设计技术与应用指南。

---

## 一、等价类划分法

### 定义
将输入域划分为若干等价类，从每个等价类中选取一个代表性值作为测试数据。

### 分类
- **有效等价类**：符合需求的输入集合，程序应正常处理
- **无效等价类**：违反需求的输入集合，程序应给出错误提示

### 应用规则
1. 每个有效等价类至少覆盖一次
2. 每个无效等价类单独设计用例（避免缺陷掩盖）
3. 覆盖所有输入条件（显式和隐式）

### 常见等价类划分维度
| 输入类型 | 有效等价类示例 | 无效等价类示例 |
|---------|-------------|-------------|
| 数值输入 | 范围内值 | 超范围值、非数值、空值 |
| 字符串输入 | 合法字符组合 | 特殊字符、超长字符串、空串 |
| 日期输入 | 合法日期 | 2月30日、格式错误、未来日期 |
| 枚举选择 | 列表内值 | 列表外值、空选择 |
| 文件上传 | 支持格式 | 不支持格式、超大文件、空文件 |

## 二、边界值分析法

### 定义
在等价类边界及其附近选取测试数据，因为边界是缺陷高发区。

### 基本原则
- 对于范围 [a, b]，测试点取：a-1, a, a+1, b-1, b, b+1
- 对于离散值集合，测试集合的首尾值
- 对于有序集合，测试第1个、第2个、倒数第2个、最后1个

### 常见边界场景
| 场景 | 边界值 |
|------|-------|
| 输入长度 | 最小长度-1、最小长度、最小长度+1、最大长度-1、最大长度、最大长度+1 |
| 数值范围 | 下限-1、下限、下限+1、上限-1、上限、上限+1 |
| 列表分页 | 第0页、第1页、最后一页、超出总页数 |
| 日期范围 | 起始日-1天、起始日、结束日、结束日+1天 |
| 文件大小 | 0KB、最小值、最大值、超出最大值 |

## 三、场景法

### 定义
通过分析软件的业务流程（基本流和备选流），构造场景进行测试。

### 基本流程分析
1. **基本流（Basic Flow）**：用户最常走的"快乐路径"
2. **备选流（Alternative Flow）**：基本流的分支变体
3. **异常流（Exception Flow）**：错误和异常处理路径

### 场景构造矩阵
| 场景编号 | 流程组合 | 预期结果 |
|---------|--------|--------|
| 场景1 | 基本流 | 正常完成 |
| 场景2 | 基本流 + 备选流1 | 分支路径1处理 |
| 场景3 | 基本流 + 异常流1 | 错误处理1 |
| 场景N | 基本流 + 异常流M | 错误处理M |

## 四、判定表法

### 适用场景
当输入条件之间存在逻辑组合关系时，使用判定表确保所有组合覆盖。

### 构建步骤
1. 列出所有条件桩（输入条件）
2. 列出所有动作桩（输出动作）
3. 填写条件项（条件的取值组合）
4. 填写动作项（每种组合对应的输出）
5. 简化合并（消除不可能的组合）

---

## 五、状态迁移法

### 适用场景
被测对象存在明确的状态转换逻辑时使用。

### 测试策略
1. 绘制状态迁移图
2. 列出所有状态和触发事件
3. 测试所有合法状态转换
4. 测试非法状态转换（不应发生的转换）
5. 测试初始状态和终态

---

## 六、错误推测法

### 定义
基于经验和直觉推测可能存在的错误，补充设计测试用例。

### 常见易错场景
- 输入为空 / 全空格
- 输入全0 / 全1 / 负数
- 日期跨年 / 跨月 / 跨时区
- 并发操作 / 重复提交
- 超时 / 网络中断后恢复
- 数据删除后关联引用
- 权限提升 / 越权访问

---

## 七、非功能性测试要点

### 性能测试
- 响应时间：关键操作响应时间是否满足要求
- 并发能力：系统支持的并发用户数
- 吞吐量：单位时间处理请求数
- 资源占用：CPU、内存、磁盘IO使用率
- 长时间运行：内存泄漏、连接池耗尽

### 安全性测试
- 输入验证：SQL注入、XSS、命令注入
- 认证授权：越权访问、会话管理、密码策略
- 数据保护：敏感数据加密、传输安全、存储安全
- 接口安全：API认证、频率限制、参数篡改

### 兼容性测试
- 浏览器兼容：Chrome/Firefox/Safari/Edge
- 设备兼容：PC/平板/手机
- 系统兼容：Windows/macOS/iOS/Android
- 分辨率适配：不同屏幕尺寸
- 网络环境：WiFi/4G/5G/弱网

## 八、优先级判定标准

### P0 - 必须通过
- 阻塞主流程的核心功能
- 影响核心业务逻辑
- 无替代方案的严重问题
- 安全漏洞类问题

### P1 - 应当通过
- 重要功能异常但有替代方案
- 影响用户体验的问题
- 非核心流程的完整性问题
- 数据一致性问题

### P2 - 可以延后
- 边缘场景
- UI细节问题
- 低概率异常
- 体验优化类问题
