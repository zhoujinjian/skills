# 分类分级判定规则

本文档定义 review-testcase 技能的三级分类判定标准、否决规则和判定流程。

---

## 三级分类定义

| 分类 | 代号 | 含义 | 后续动作 |
|------|------|------|---------|
| 可用 | L1 | 标准正向/反向用例，可直接入库 | 进入测试执行 |
| 待修改 | L2 | 逻辑方向对，但描述或数据需要调整 | 标注修改点后返回修改 |
| 错误无效 | L3 | 业务逻辑错误、与需求不符、无法执行 | 直接废弃 |

---

## 判定逻辑：评分驱动 + 否决规则修正

分类不是凭感觉，而是以5维评分为基础，叠加关键否决规则进行综合判定。

### 第一类（可用）：可直接入库

**必须同时满足以下所有条件：**

| 条件类型 | 具体标准 |
|---------|---------|
| 总分条件 | 5维总分 ≥ 80分（满分100） |
| 维度下限 | 每个维度得分 ≥ 该维度满分的60% |
| 维度下限具体值 | 逻辑完整性 ≥ 15，预期结果明确性 ≥ 12，前置条件完备性 ≥ 9，PRD覆盖度 ≥ 15，边界异常覆盖 ≥ 9 |
| 业务逻辑 | 步骤逻辑与PRD需求一致，无矛盾 |
| 可执行性 | 测试步骤可按原文直接执行，无需推断或补充 |
| 预期结果 | 预期结果具体可验证（含具体数值/状态/页面元素） |
| 非重复 | 与其他用例无语义重复 |

### 第二类（待修改）：逻辑方向对，需调整

**满足以下任一条件：**

| 条件类型 | 具体标准 |
|---------|---------|
| 总分条件 | 5维总分在 50-79 分之间 |
| 或：高分但维度不均 | 总分 ≥ 80 但存在 1-2 个维度低于该维度满分的60% |
| 可修复性 | 逻辑方向正确（测的是对的功能点），通过补充/修改可修复 |

**典型待修改场景：**

1. **预期结果笼统**：如"验证成功"无具体判定标准 → 需补充具体判定条件
2. **前置条件缺1-2项**：如未说明账号权限 → 需补充权限说明
3. **步骤有1处跳跃**：如缺少"点击确认按钮" → 需补充中间步骤
4. **测试数据为占位符**：如"xxx""测试数据""合法值" → 需替换为具体数据
5. **边界值部分缺失**：测了正常值但遗漏1-2个边界 → 需补充边界用例
6. **预期结果缺少验证维度**：只验证了页面行为，未验证数据状态 → 需补充

### 第三类（错误无效）：直接废弃

**满足以下任一条件：**

| 条件类型 | 具体标准 |
|---------|---------|
| 总分条件 | 5维总分 < 50 分 |
| 维度严重不足 | 存在 3 个及以上维度低于该维度满分的60% |
| 否决规则命中 | 任一否决规则命中（见下方否决规则清单） |

## 否决规则清单（任一命中即归为"错误无效"）

否决规则是硬性判定，无论总分高低，只要命中任一规则即归入第三类。

### 否决规则1：业务逻辑与PRD矛盾

**判定标准**：用例的业务逻辑与PRD需求文档中的规则明确矛盾

**判定方式**：
- 需要PRD文档作为参考
- AI对比用例步骤与PRD规则，发现明确矛盾
- 示例：PRD要求"满100减20"，用例验证"满100减10"

**无需PRD时的替代判定**：当无法获取PRD时，本规则不单独触发，但可通过其他否决规则覆盖。

### 否决规则2：测试不存在的功能

**判定标准**：用例描述的功能在系统中不存在

**判定方式**：
- AI根据PRD或系统常识判断
- 示例：用例测试"一键分享到朋友圈"，但系统根本无此功能

### 否决规则3：步骤逻辑自相矛盾无法执行

**判定标准**：步骤之间存在逻辑矛盾，按原文无法执行

**判定方式**：
- 分析步骤间的因果关系
- 示例："1. 在未登录状态下访问个人中心 2. 查看订单列表"——未登录无法查看订单

### 否决规则4：与其他用例完全重复且无增量价值

**判定标准**：与另一条用例的测试步骤、测试数据、预期结果完全一致（或仅编号/标题不同）

**判定方式**：
- 语义相似度比对：步骤+数据+预期结果的核心语义一致
- 需判断是否有增量价值（如同一操作不同数据不算重复）

**非重复的例外情况**：
- 同一操作不同测试数据（如正常值和边界值）
- 同一功能不同前置条件（如管理员和普通用户）
- 正向和反向用例测同一功能点

### 否决规则5：预期结果与业务规则完全不符

**判定标准**：预期结果描述与已知的业务规则完全相反

**判定方式**：
- AI根据PRD或系统常识判断
- 示例：预期结果写"余额不足时支付成功"，但业务规则明确余额不足应拒绝支付

## 判定流程图

```
逐条用例评审
   │
   ├── 硬性检查（结构化规则）
   │   ├── 预期结果为空？ ──────────── 是 → 第三类（错误无效）
   │   ├── 步骤少于2步？ ──────────── 是 → 第二类（待修改），逻辑完整性扣分
   │   ├── 预期结果含模糊词？ ──────── 是 → 第二类（待修改），预期结果明确性扣分
   │   └── 测试数据含占位符？ ──────── 是 → 第二类（待修改），相关维度扣分
   │
   ├── 否决规则检查
   │   ├── 规则1: 业务逻辑与PRD矛盾？ ── 是 → 第三类（错误无效）
   │   ├── 规则2: 测试不存在的功能？ ── 是 → 第三类（错误无效）
   │   ├── 规则3: 步骤自相矛盾？ ────── 是 → 第三类（错误无效）
   │   ├── 规则4: 完全重复？ ────────── 是 → 第三类（错误无效）
   │   └── 规则5: 预期结果与业务规则相反？── 是 → 第三类（错误无效）
   │
   ├── 评分阈值检查
   │   ├── 总分 < 50？ ────────────── 是 → 第三类（错误无效）
   │   ├── 3个及以上维度 < 60%？ ──── 是 → 第三类（错误无效）
   │   ├── 总分 ≥ 80 且各维度 ≥ 60%？── 是 → 第一类（可用）
   │   ├── 总分 50-79？ ───────────── 是 → 第二类（待修改）
   │   └── 总分 ≥ 80 但有1-2个维度 < 60%？── 是 → 第二类（待修改）
   │
   └── 输出分类结果 + 推理链
```

## 结构化规则（确定性硬检查）

以下规则是硬性的，不需要AI"理解"，只需要"检查"。在AI语义推理之前先执行这些检查：

| 检查项 | 检查方式 | 命中则 | 影响维度 |
|--------|---------|--------|---------|
| 预期结果是否为空 | 字段为空/仅含空白字符检测 | 直接归为第三类 | - |
| 步骤是否少于2步 | 数量检测（以编号分隔计数） | 归为第二类 | 逻辑完整性-5分 |
| 预期结果含模糊词 | 关键词匹配（验证成功、功能正常等） | 归为第二类（若仅此问题） | 预期结果明确性-3分/处 |
| 测试数据含占位符 | 关键词匹配（xxx、测试数据、合法值、任意有效） | 归为第二类 | 相关维度扣分 |
| 前置条件为空 | 字段为空/仅含"无"/空白字符 | 扣分 | 前置条件完备性-6分 |
| 用例标题与内容不符 | 语义推理 | 扣分 | 逻辑完整性-3分 |

### 模糊预期结果关键词库

```
验证成功、功能正常、页面正确、操作成功、系统正常、显示正确、
运行正常、正常显示、正常工作、功能可用、页面正常、响应正确、
处理成功、提交成功、保存成功
```

### 占位符关键词库

```
xxx、XXX、测试数据、合法值、任意有效、正常数据、有效输入、
正确数据、合适的值、符合要求的、合规数据、TestData
```

## 语义理解规则（AI推理判断）

以下规则需要AI结合PRD和业务知识进行推理，每条推理结果必须输出推理链：

| 判断维度 | AI推理依据 | 输出形式 |
|---------|-----------|---------|
| 业务逻辑是否正确 | 对照PRD需求描述，验证步骤是否符合业务规则 | 判定结果 + 依据引用（PRD位置） |
| 步骤是否有跳跃 | 分析步骤间的因果链，检测是否有缺失中间操作 | 跳跃点标注（哪两步之间缺少什么） |
| 预期结果是否可验证 | 判断预期结果是否包含可观测、可度量的判定标准 | 不可验证项标注 + 补充建议 |
| 前置条件是否完备 | 分析步骤依赖，反推需要哪些前置条件 | 缺失项列表 |
| 是否语义重复 | 语义相似度比对，判断不同用例是否测试同一场景 | 重复组标注（列出重复的用例编号） |
| 边界异常是否充分 | 分析输入项和数据范围，判断是否有遗漏 | 遗漏项列表 |

### 推理链输出格式

```
推理链:
  1. [逻辑完整性-3] 第3步到第4步存在跳跃，缺少"点击确认按钮"的中间操作
  2. [预期结果-4] 第2步预期结果为"验证支付成功"，未说明判定标准（应明确：订单状态变为"已支付"，扣款金额=99.00元）
  3. [前置条件-4] 未说明订单支付状态（应补充"已有待支付订单"）
  4. [边界异常-4] 未测试金额为0的边界场景
  5. [PRD覆盖-5] PRD要求支持"优惠券叠加"，本用例未覆盖此场景
分类: 待修改（总分68，否决规则未命中，但总分<80）
```

## 评分与分类的对应关系速查表

| 5维总分 | 维度最低比例 | 否决规则 | 最终分类 |
|--------|------------|---------|---------|
| ≥ 80 | 全部 ≥ 60% | 未命中 | ✅ 可用 |
| ≥ 80 | 1-2个 < 60% | 未命中 | ⚠️ 待修改 |
| ≥ 80 | 3+个 < 60% | 未命中 | ❌ 错误无效 |
| 50-79 | - | 未命中 | ⚠️ 待修改 |
| 50-79 | - | 命中 | ❌ 错误无效 |
| < 50 | - | 未命中 | ❌ 错误无效 |
| 任意 | - | 命中 | ❌ 错误无效 |
