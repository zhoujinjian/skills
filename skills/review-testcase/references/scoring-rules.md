# 5维评审打分细则

本文档为 review-testcase 技能提供详细的5维评审打分标准。每个维度包含分段评分标准、扣分项清单和扣分分值。

---

## 维度1：逻辑完整性（满分 25 分）

**评审重点**：步骤是否清晰、逻辑链路是否跑通、是否存在跳跃或矛盾

### 分段评分标准

| 得分区间 | 判定标准 | 典型特征 |
|---------|---------|---------|
| 22-25 | 步骤完整无跳跃，因果链路清晰，从前置到验证形成闭环 | 每步之间有明确因果，操作→响应→下一步逻辑连贯 |
| 18-21 | 步骤基本完整，有1处小跳跃（如跳过登录步骤），但不影响理解 | 缺少1个中间操作，但读者可推断 |
| 14-17 | 有2-3处跳跃，需读者自行推断中间步骤 | 多处缺少衔接，需结合上下文猜测 |
| 8-13 | 逻辑链路有明显断裂，步骤之间缺少衔接 | 步骤间因果关系不清，无法直接执行 |
| 0-7 | 步骤混乱、逻辑矛盾、无法理解执行流程 | 步骤顺序颠倒、自相矛盾、无法执行 |

### 扣分项清单

| 扣分项 | 扣分值 | 判定方式 | 示例 |
|--------|--------|---------|------|
| 步骤跳跃（缺少关键中间操作） | -3分/处 | 结构化检查：步骤间缺少必要衔接 | "1. 打开支付页面 2. 支付成功"——缺少选择支付方式、确认支付 |
| 步骤顺序颠倒导致逻辑不通 | -5分 | 语义推理：结果在原因之前 | "1. 验证订单状态为已支付 2. 点击支付按钮" |
| 操作与预期结果不对应 | -5分 | 结构化检查：步骤无对应预期 | 步骤5的操作在预期结果中无对应验证 |
| 步骤描述模糊无法执行 | -3分/处 | 关键词匹配："进行操作""执行测试" | "1. 进行正常操作 2. 验证功能正常" |
| 步骤少于2步 | -5分 | 数量检测 | 整个用例只有1步操作 |
| 循环引用或自相矛盾 | -8分 | 语义推理 | 步骤3依赖步骤5的结果 |

## 维度2：预期结果明确性（满分 20 分）

**评审重点**：每一步的预期结果是否可验证，金额、状态是否有明确定义

### 分段评分标准

| 得分区间 | 判定标准 | 典型特征 |
|---------|---------|---------|
| 18-20 | 每步预期结果可验证，金额/状态/页面元素有具体值 | "订单状态变为'已支付'，扣款金额=99.00元，页面跳转到支付成功页" |
| 14-17 | 大部分明确，1-2处使用模糊表述 | 多数预期有具体值，个别用"验证成功"等模糊词 |
| 10-13 | 预期结果存在但多处不够具体，缺乏判定依据 | "页面显示正确""功能正常运行" |
| 5-9 | 预期结果笼统，无法直接验证 | "验证成功""功能正常""页面正确" |
| 0-4 | 缺少预期结果或预期结果与步骤无关 | 空白、只有"无"、与步骤完全无关 |

### 扣分项清单

| 扣分项 | 扣分值 | 判定方式 | 示例 |
|--------|--------|---------|------|
| 使用"验证成功""功能正常"等模糊词 | -3分/处 | 关键词匹配：验证成功、功能正常、页面正确、操作成功、系统正常 | "验证登录成功" |
| 金额/数量未给出具体值 | -4分/处 | 语义推理：预期结果涉及金额/数量但无具体数值 | "扣款金额正确"——缺少具体金额 |
| 状态判定缺少期望值 | -3分/处 | 语义推理：检查状态但未说明期望值 | "检查订单状态"——未说明期望"已取消" |
| 预期结果缺少页面行为验证 | -2分/处 | 结构化检查：无页面行为描述 | 只写了数据状态，没有页面跳转/提示信息 |
| 预期结果缺少数据状态验证 | -2分/处 | 结构化检查：无数据状态描述 | 只写了页面提示，没有数据/接口层面验证 |
| 预期结果为空 | -8分 | 字段为空检测 | 预期结果字段无内容 |

### 模糊预期结果关键词库（用于自动检测）

```
验证成功、功能正常、页面正确、操作成功、系统正常、显示正确、
运行正常、正常显示、正常工作、功能可用、页面正常、响应正确、
处理成功、提交成功、保存成功（无具体状态描述时）
```

## 维度3：前置条件完备性（满分 15 分）

**评审重点**：环境、数据、权限、商品类型等前置条件是否完整

### 分段评分标准

| 得分区间 | 判定标准 | 典型特征 |
|---------|---------|---------|
| 13-15 | 环境、账号权限、测试数据、商品/订单类型等前置条件完整 | "已登录普通用户账号，购物车中有1件商品（iPhone 15，单价5999元），账户余额10000元" |
| 10-12 | 基本完整，缺少1项非关键前置条件 | 有账号和数据，但未说明浏览器版本 |
| 7-9 | 缺少2-3项前置条件，需读者自行补充 | 有账号但未说明权限和数据状态 |
| 3-6 | 前置条件严重不足，缺少关键数据或权限说明 | 只写"登录系统"，未说明账号类型和数据 |
| 0-2 | 无前置条件或前置条件与用例无关 | 空白、"无"、"系统已部署" |

### 扣分项清单

| 扣分项 | 扣分值 | 判定方式 | 示例 |
|--------|--------|---------|------|
| 未说明账号权限/角色 | -3分 | 语义推理：涉及权限操作但未说明角色 | 支付用例未说明用管理员还是普通用户 |
| 未说明测试数据状态 | -4分 | 语义推理：依赖特定数据但未说明 | 订单取消用例未说明订单当前状态 |
| 未说明环境要求 | -2分 | 结构化检查：涉及特定环境但未说明 | 需要特定支付渠道但未说明 |
| 前置条件为空 | -6分 | 字段为空检测 | 前置条件字段无内容 |
| 前置条件与用例无关 | -4分 | 语义推理：前置条件无法支撑步骤 | 前置条件写"系统已部署"，但用例需要特定数据 |

### 需要前置条件的信号词（检测缺失）

```
当用例步骤/标题中出现以下关键词，但前置条件中未说明对应条件时，视为缺失：
支付 → 需要说明：账号余额/支付方式/订单状态
取消 → 需要说明：当前状态（如"待支付"）
删除 → 需要说明：存在的数据项
修改/编辑 → 需要说明：已有数据+当前值
查询/搜索 → 需要说明：预期匹配的数据已存在
上传 → 需要说明：文件类型/大小限制
导出 → 需要说明：数据量/格式
权限 → 需要说明：角色/权限配置
```

## 维度4：PRD覆盖度（满分 25 分）

**评审重点**：是否覆盖了需求文档中的核心功能点与联动规则

### 分段评分标准

| 得分区间 | 判定标准 | 典型特征 |
|---------|---------|---------|
| 22-25 | 完整覆盖需求文档核心功能点与联动规则，无遗漏 | 用例与PRD功能点一一对应，联动规则有独立用例 |
| 18-21 | 覆盖核心功能点，遗漏1个非核心联动规则 | 主流程和分支都覆盖，但个别组合规则遗漏 |
| 14-17 | 覆盖主要功能点，遗漏联动规则或特殊场景 | 主功能覆盖，但联动规则或特殊条件未覆盖 |
| 8-13 | 仅覆盖部分功能点，遗漏核心逻辑 | 有明显功能点遗漏 |
| 0-7 | 与需求严重不符或遗漏核心功能 | 测试的功能在PRD中不存在，或遗漏核心流程 |

### 扣分项清单

| 扣分项 | 扣分值 | 判定方式 | 示例 |
|--------|--------|---------|------|
| 未覆盖PRD明确要求的子场景 | -5分/个 | 需PRD文档对照 | PRD要求"支持微信和支付宝两种支付方式"，用例只测了微信 |
| 业务规则理解错误 | -10分 | 需PRD文档对照 | PRD要求"满100减20"，用例验证"满100减10" |
| 遗漏核心联动逻辑 | -8分 | 需PRD文档对照 | PRD要求"A条件触发B操作"，用例未覆盖 |
| 与PRD功能点不对应 | -10分 | 需PRD文档对照 | 用例测试了PRD中不存在的功能 |

### 无PRD文档时的处理

当用户未提供PRD文档时，本维度评分采用"自身完整性评估"策略：
- 25分：用例覆盖了从输入到输出的完整业务链路
- 20分：覆盖主流程，个别分支遗漏
- 15分：覆盖主要路径，异常/分支覆盖不足
- 10分：覆盖部分路径，有明显遗漏
- 5分：仅覆盖单一场景
- 0分：无法判断与需求的对应关系

同时在评审结果中标注："⚠️ 未提供PRD文档，本维度采用自身完整性评估，分数仅供参考"

## 维度5：边界异常覆盖（满分 15 分）

**评审重点**：边界值、并发、互斥、错误处理是否充分

### 分段评分标准

| 得分区间 | 判定标准 | 典型特征 |
|---------|---------|---------|
| 13-15 | 边界值、并发、互斥、错误处理场景充分 | 每个输入项有边界测试，异常路径有独立用例 |
| 10-12 | 边界值覆盖较好，异常场景略有不足 | 主要边界值已测试，部分异常场景缺失 |
| 7-9 | 有部分边界值测试，异常场景不足 | 仅测试了部分边界，异常处理不充分 |
| 3-6 | 仅有1-2个边界/异常场景 | 几乎只有正常流程 |
| 0-2 | 无边界和异常覆盖，仅正常流程 | 纯正向用例，无任何异常和边界测试 |

### 扣分项清单

| 扣分项 | 扣分值 | 判定方式 | 示例 |
|--------|--------|---------|------|
| 数值边界未测试（0、负数、最大值） | -3分/处 | 语义推理：涉及数值输入但无边界用例 | 金额输入未测试0元和最大金额 |
| 空值/null/空字符串未测试 | -2分/处 | 语义推理：必填字段无空值用例 | 用户名未测试空输入 |
| 特殊字符未测试 | -2分/处 | 语义推理：文本输入无特殊字符用例 | 搜索框未测试SQL注入和XSS |
| 并发/竞态场景缺失 | -4分 | 语义推理：涉及共享资源但无并发用例 | 支付未测试同一订单并发支付 |
| 错误处理路径缺失 | -3分 | 语义推理：有异常可能但无错误处理用例 | 网络超时、服务不可用等未测试 |
| 互斥条件未测试 | -3分 | 语义推理：有互斥规则但无对应用例 | 优惠互斥规则未测试 |

### 边界异常场景识别信号词

```
当用例/需求中出现以下关键词时，应检查是否有对应的边界/异常用例：
金额/价格/数量/长度 → 边界值测试
必填/不能为空 → 空值测试
唯一/不可重复 → 重复值测试
同时/并发/并发数 → 并发测试
互斥/不能同时 → 互斥测试
超时/失败/异常 → 错误处理测试
权限/角色/授权 → 越权测试
```

## 评分一致性原则

1. **同类问题同分**：相同类型的问题在不同用例中应扣除相同分数
2. **扣分累加不超上限**：单个维度的扣分总额不超过该维度满分
3. **最低0分**：任何维度得分不低于0分
4. **整数评分**：所有维度评分取整数
5. **扣分必须注明位置**：扣分必须引用用例中的具体内容位置（如"第3步""前置条件""预期结果第2条"）
